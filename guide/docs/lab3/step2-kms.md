# Step 2: Create the Key Store

AWS KMS provides the root of our key hierarchy. The KMS key (Key Encryption Key / KEK) wraps and unwraps the Data Encryption Keys (DEKs) that protect individual TDF files. The KEK itself never leaves the KMS hardware security module.

## Why KMS?

- **Hardware-backed**: Keys are stored in FIPS 140-3 Level 3 validated HSMs
- **Audit logged**: Every use of the key is recorded in CloudTrail
- **Access controlled**: Key policies define exactly which IAM roles can use the key
- **Rotation**: Automatic annual key rotation available

This mirrors the hardware security modules (HSMs) used in defence environments for key management.

## Create the KMS key

1. Go to **KMS Console**: [https://console.aws.amazon.com/kms](https://console.aws.amazon.com/kms)
2. Click **Create key**
3. **Key type**: Symmetric
4. **Key usage**: Encrypt and decrypt
5. Click **Next**
6. **Alias**: `dcs-level3-kas-kek`
7. **Description**: `Key Encryption Key for DCS Level 3 KAS - wraps TDF Data Encryption Keys`
8. Click **Next**
9. **Key administrators**: Select your IAM user/role
10. Click **Next**
11. **Key usage permissions**: We'll update this later to add the ECS task role. For now, select your IAM user/role.
12. Click **Next** > **Finish**

## Note the Key ID

On the key details page, copy the **Key ID** (a UUID like `12345678-abcd-1234-efgh-123456789012`). You'll need this when configuring the OpenTDF platform.

## Understand the key hierarchy

```
AWS KMS Key (KEK)
  "dcs-level3-kas-kek"
  - Never leaves KMS HSM
  - FIPS 140-3 Level 3
  - Every use logged in CloudTrail
       |
       | Wraps (encrypts)
       v
Data Encryption Key (DEK) #1 -----> Encrypts intel-report.tdf payload
Data Encryption Key (DEK) #2 -----> Encrypts wall-report.tdf payload
Data Encryption Key (DEK) #3 -----> Encrypts logistics.tdf payload
...
```

Each TDF file has its own unique DEK. The DEK is generated by the OpenTDF SDK, used to encrypt the data payload (AES-256-GCM), and then wrapped by KMS. The wrapped DEK is stored in the TDF manifest. To decrypt, the KAS sends the wrapped DEK to KMS for unwrapping, but only after verifying the user's attributes.

## Why this matters for DCS

The KMS key is the trust anchor. If you control the KMS key, you control who can decrypt data. In a coalition scenario:

- **Each nation could have their own KMS key** in their own AWS account
- **A TDF file can reference multiple KAS/KMS combinations** (one per nation)
- **Any nation's KAS can independently unwrap** if the TDF uses the "AnyOf" pattern
- **No nation needs access to another's KMS key** - each wraps independently

This is how federated key management works: shared data, independent keys, sovereign control.
